1. Overview

The PUG Legal Case Control System centralizes the end-to-end legal case lifecycle for Paris United Group's multi-branch divisions — from case intake at division accounts, audit review, FM/ED/MD/Chairman approvals, court filing by the lawyer, hearing tracking, expense management and final closure — with role-based access, branded notifications, dashboards and full audit trails.

This document is the canonical phase-by-phase delivery plan. Phases are sized to fit within a single Claude Code 5-hour token window so work does not get interrupted mid-feature. Coding begins only after the user types start Phase 0.

2. Technology Stack

Backend

  • Python 3.12
  • FastAPI (async REST + OpenAPI)
  • SQLAlchemy 2.x (ORM) + Alembic migrations
  • Pydantic v2 for validation
  • Celery + Redis (jobs & schedules)
  • APScheduler for cron reports

Frontend

  • Next.js 14 (App Router) + TypeScript
  • React 18, TanStack Query
  • Tailwind CSS + shadcn/ui
  • Recharts / ApexCharts
  • Zustand for client state
  • Light / Dark mode tokens

Database & Cache

  • PostgreSQL 16
  • Redis 7 (cache, queue, sessions)
  • S3-compatible object storage for case docs

Documents & Reports

  • WeasyPrint / ReportLab (PDF)
  • OpenPyXL + XlsxWriter (Excel)
  • Jinja2 branded templates
  • Pillow for image & signature processing

Security & Auth

  • JWT (access + refresh)
  • RBAC + division scoping
  • Argon2 password hashing
  • 2FA via TOTP (optional)
  • Rate-limit & CSRF

DevOps

  • Direct local dev (PugFin-style)
  • systemd units + nginx (at deploy time)
  • Docker / docker-compose files kept ready for future cloud deploys
  • GitHub Actions CI
  • Sentry / Loguru
  • pgBackRest scheduled DB backups

3. Full Feature List

3.1 Core Modules

  • Legal Case Entry Form (multi-cheque, attachments, signatories)
  • Multi-stage approval workflow (Accountant → Manager → Auditor → FM → ED → MD/Chairman → Lawyer)
  • Court filing module with Police Case No., Court Case No., filed date, govt acknowledgment
  • Hearing tracker (next hearing date/time/location, command updates, attachments)
  • Lawyer cash-request & expense ledger per case
  • Case closure with multi-status (Closed / Rejected / Fund Received / Pending / Hold / Signatory Out of Country)

3.2 Reporting & Export

  • Export Excel export on every list/report (XLSX with branded header, frozen rows)
  • Export PDF export with PUG branded header/footer, watermarks, page numbers
  • Print Direct browser print with print-optimized CSS (corporate layout)
  • Case-flow chart printable per case (visual lifecycle)
  • Detailed case report including all attachments preview
  • Aging report, recovery report, division-wise summary, status pivot

3.3 Backup & Restore PugFin Parity

  • Same engine as PugFin App: on-demand & scheduled DB dumps (pg_dump + custom format)
  • Attachment archive (tar.gz of object storage namespace)
  • Encrypted backups (AES-256 with system key) stored locally + optional S3 offsite
  • Retention policy (daily 7 / weekly 4 / monthly 12)
  • Restore wizard with pre-restore safety snapshot
  • Backup integrity check (checksum + test-restore job)
  • Audit log entry for every backup & restore action

3.4 Email Log

  • Every outgoing email recorded: To/CC/BCC, subject, template, status (Queued/Sent/Bounced/Failed), timestamp, attempts, related case & user
  • Searchable, filterable; preview rendered email (HTML) and download attachments
  • Resend / cancel queued mails from UI
  • SMTP failure alerting

3.5 Audit Log

  • Append-only log of every create / update / delete / approve / reject / status-change / login / failed-login / permission change
  • Captures: actor, role, IP, user-agent, before-value, after-value, entity, entity ID, reason
  • Filter by user, module, division, date range; export to Excel/PDF
  • Tamper-evident chain (row hash linked to previous row)

3.6 Scheduled Reporting via Email Branded

  • Build any report → save as scheduled job (daily/weekly/monthly/cron expression)
  • Choose recipients (users, roles, custom emails), CC, BCC
  • Choose format(s): PDF, Excel, inline HTML summary
  • Branded email template (PUG logo, gold/navy theme, footer signature block)
  • Per-schedule run history, last-run status, next-run countdown
  • Pause / resume / run-now controls

1. AccountantFill Legal Entry Form, attach docs, submit, print
2. Sales ManagerReview & sign
3. Division ManagerApprove, send physical docs to HO
4. AccountantStatus: Sent to Head Office
5. HO AuditReceive physical → Application Received
6. AuditorVerify vs accounting system → Handover to FM
7. FMReview, request clarification or forward to ED
8. EDSign physical, status update
9. Chairman / MDFinal approval
10. FM → LegalHand over with command
11. LawyerFile case in govt system, upload Acknowledgement, Police No., Court No.
12. Lawyer / FMHearing updates, next dates, attachments
13. LawyerCash request → FM approve → Accountant pay → Receipt
14. FMFinal status: Closed / Rejected / Fund Received / Hold / Pending

Phase 7 ✓Scheduled Reporting via EmailCompleted

  • Migration 0006_phase7_scheduled_reports; tests for CRUD, bad cron, unknown report, run-now history, pause/resume, delete
Phase 8 ✓Audit Log (Tamper-Evident)Completed
  • AuditLog table with prev_hash + row_hash forming a SHA-256 chain; rows are append-only and the tail is row-locked while writing
  • Per-request context (IP + User-Agent) carried via contextvars; the JWT dep attaches actor (id / email / role) so every audit row is fully attributed
  • Explicit audit_service.record_event() calls at all key mutation points: login / login_failed / password_change, master CRUD (Division, Bank, Customer, Salesman, Lawyer, Case Type), user + role CRUD, case create / delete, every workflow transition (submit / approve / reject / clarify / resubmit), court filing, hearings, cash request create / approve / reject / pay
  • /api/v1/audit-log endpoints: list with filters (action, entity, actor, date range, text search), detail with before/after JSON + hashes, verify walks the chain and reports broken links, .csv and .pdf exports
  • UI at /admin/audit-log: filter bar, paginated table, click-to-open detail drawer with field-level diff (before in red, after in green), CSV / PDF download buttons, big "Verify Chain" button that surfaces tampering in a red banner
  • Migration 0007_phase8_audit; tests for login audit, master CRUD audit, chain verify, tamper detection, CSV/PDF export and filter
Phase 9 ✓Backup & Restore (PugFin Parity)Completed
  • BackupJob + RestoreJob models record every operation with size, SHA-256 checksum, table row counts, attachment count and full manifest
  • Backup engine snapshots every table (SQLAlchemy-driven, dialect-portable) plus the entire storage/ attachments tree, packaged as a single tar.gz bundle
  • AES-256-GCM envelope via cryptography: per-file 12-byte nonce + GCM tag; key from BACKUP_ENCRYPTION_KEY (base64-encoded 32 bytes); plain mode when unset
  • Verify endpoint re-checksums the on-disk file, decrypts (when applicable) and confirms manifest.json + data.json are present
  • Restore wizard: optional auto pre-restore safety snapshot, requires typing RESTORE, wipes every table in reverse FK order then replays the bundle, then resets Postgres sequences
  • Sidebar Backup & Restore page: KPI strip (count / size / last backup / encryption), Create Backup button with notes, table with Verify / Download / Restore / Delete actions, rose-bordered restore modal
  • All actions audited (create / download / delete / restore) via the Phase 8 chain
  • Migration 0008_phase9_backup; tests for encrypted + plaintext backups, verify, restore round-trip, confirmation requirement, delete, and status endpoint
  • Scheduled backups, retention policies and S3 offsite land in Phase 10 (Admin Settings)
Phase 10 ✓System Settings & Admin ConsoleCompleted
  • SettingsKV table with one row per setting; sensitive fields stored as ENC:<base64> AES-256-GCM envelopes (reuses the Phase 9 crypto key)
  • Static descriptor of all 12 setting groups (Company & Branding, SMTP, AI, Numbering, Workflow, Security, Backup, Notifications, Appearance, Integrations, Data Management, Maintenance) with per-field type metadata + env fallback name
  • DB values override env vars at runtime; the SMTP backend now calls settings_service.effective_smtp() per-send so changes take effect immediately
  • API endpoints: GET /settings/groups, GET/PUT /settings/groups/{key}, POST /settings/smtp/test-send (sends a branded notification via the live SMTP config)
  • Every settings save and SMTP test is recorded in the Phase 8 audit chain
  • /admin/settings: tabbed UI with one tab per group, dynamic field rendering (text / textarea / password / number / checkbox / color / select / url / email), masked sensitive values, source badge (db / env / default), Save button counter for dirty fields, Test Send button on the SMTP group
  • /admin/diagnostics: live health for Database, Redis, Phase 7 Scheduler, last backup and the backup encryption key; KPI strip + colour-coded rows
  • Migration 0009_phase10_settings; tests for descriptor coverage (all 12 groups), SMTP round-trip with masked password, sensitive value encryption at rest, rejected unknown field, SMTP test send in console mode and the diagnostics endpoint
  • Sidebar Admin group drops the "(Phase 10)" suffix and adds Health & Diagnostics
